# 少女智能体 GirlAgent 设计要点（V0 框架阶段）

## 0. 核心准则（最高优先级）

GirlAgent 的最高优先级准则：

- 能连接任何 AI 应用（云端 API、本地模型网关、自建服务、第三方聚合平台）
- 以“功能丰富的少女智能体”为目标形态，而非单一聊天壳
- 同时提供应用版与无头版，在有桌面和无桌面环境都可用

其中“能连接任何 AI 应用”通过统一协议适配层实现，而不是为某一家模型厂商做硬编码。

## 1. 项目目标

GirlAgent 是一个应用端智能体构建软件，核心目标是：

- 构建高度拟人的智能体体验
- 支持多模型协同（多模态能力扩展）
- 支持多 Key 负载均衡，提升可用性和稳定性
- 兼容云端模型与本地部署模型
- 让普通用户通过配置即可完成智能体搭建并对话

V0 阶段先完成工程框架和设置页骨架，不做完整业务闭环。

## 2. 技术与工程边界

- 框架：Tauri（应用容器，当前桌面形态）
- 后端：Rust（配置管理、模型路由、提示词组装、工具编排）
- 前端：React + TypeScript + Vite（配置界面与交互）
- 目录策略：前后端分离，根目录保持简洁

目录约定：

```text
GirlAgent/
├─ docs/
└─ apps/
   ├─ web/
   │  ├─ server/         # 无头版宿主（HTTP）
   │  └─ console/        # 共享配置界面
   └─ app/               # 应用版宿主（当前基于 Tauri）
```

Core 作为独立仓库维护：

- `git@github.com:KurohaneKaoruko/Girl-Agent-Core.git`

## 3. 统一模型接入策略（云端 + 本地）

### 3.1 模型类型覆盖目标

项目需要支持本地部署与云端调用的多类模型：

- 语言模型（LLM）
- 视觉模型（VLM / 图像理解）
- 语音识别模型（ASR / STT）
- 语音合成模型（TTS）

扩展目标（非 V0 强制）：

- 向量模型（Embedding）
- 重排模型（Reranker）
- 语音活动检测（VAD）

### 3.2 本地部署模型兼容原则

“支持所有本地部署模型”的实现原则不是硬编码每个框架，而是做统一适配层：

- 优先兼容 OpenAI 风格 API（大量本地网关可对齐该协议）
- 提供本地推理引擎适配器接口（Adapter），例如：
  - Ollama
  - LM Studio
  - llama.cpp server
  - vLLM / SGLang / TGI 等 OpenAI 兼容网关
- 允许用户自定义 API 路径映射与请求字段映射

这样可以覆盖绝大多数本地部署方案，并允许后续按插件继续扩展。

### 3.3 连接“任何 AI 应用”的协议分层

为了满足“能连接任何 AI 应用”，接口按三层抽象：

- `Transport`：HTTP / WebSocket / 本地进程调用
- `Protocol`：OpenAI 兼容、Anthropic 风格、自定义 JSON 协议
- `Capability`：文本、视觉、ASR、TTS、函数调用、流式输出

只要接入方满足其中任一协议适配器，即可被 GirlAgent 统一编排。

### 3.4 能力声明（Capability）机制

每个模型实例都应保存能力声明，而不是只存模型名：

- 输入模态：text / image / audio
- 输出模态：text / audio / tool_call
- 是否支持函数调用
- 是否支持流式输出
- 最大上下文窗口

运行时根据能力声明自动选路，避免把不支持视觉或语音的模型路由到错误任务。

## 4. 配置模型（Provider / Model / Agent）

### 4.1 模型提供商（Provider）

目标：管理 API 基地址与 Key 池。

设计要点：

- 内置常见提供商默认 API 地址（如 OpenAI、Anthropic、OpenRouter 等）
- 同时支持本地提供商项（如本地网关 `http://127.0.0.1:<port>`）
- 用户可只填写 Key，也可改写 API 地址
- 每个设置项可自定义显示名称，便于区分用途
- 同一设置项支持多个 Key（轮询/故障切换/负载均衡）
- 同一厂商可创建多个设置项（例如“主力池”“低成本池”“本地测试池”）

### 4.2 模型（Model）

目标：定义可调用模型实例与参数模板。

设计要点：

- 每个模型配置绑定一个 Provider 或使用自定义 Provider
- 必填模型 ID（云端 ID 或本地模型标识）
- 可配置参数：温度、最大 token、Top-p、频率惩罚等
- 标记模型类别：语言 / 视觉 / 语音识别 / 语音合成
- 保存能力声明（函数调用、流式、上下文窗口等）
- 支持“自定义 Provider”模式：直接填写 API + Key，不引用 Provider 设置
  - 用于“一个 Key 仅绑定一个模型”的场景

### 4.3 智能体（Agent）

目标：定义人格、行为规则与模型角色路由，不强迫用户理解底层提示词工程。

用户主要输入：

- 人格设定
- 说话规则（语气、表达方式、禁忌）

系统底层负责自动组装提示词层次：

- 背景信息（长期设定）
- 对话内容（实时上下文）
- 工具信息（检索/函数调用结果）
- 功能性系统规则（安全、格式、工具策略）

智能体模型槽位（重点）：

- 组件模型（Component）
  - 面向语音、视觉、预处理组件
  - 例如：ASR 模型、TTS 模型、图像理解模型
- 工具模型（Tool）
  - 用于工具选择、参数规划、工具结果整合
- 回复模型（Reply）
  - 用于生成最终面向用户的回复
- 决策模型（Decision）
  - 用于判断“是否需要回复”“是否触发某个动作”
  - 典型场景：桌宠常驻、QQ 机器人、被动监听类通道
  - 对话框模式通常不使用该槽位（因为对话框是必定回复）

每个槽位都可独立绑定模型并覆盖参数（温度、最大 token、超时、重试策略）。

## 5. 运行模式与决策模型策略

### 5.1 对话框模式（Chat Mode）

- 输入来自用户主动发言
- 默认必须回复
- 决策模型可关闭或跳过
- 主链路：`Reply`（可按需调用 `Tool`）

### 5.2 常驻代理模式（Ambient Mode）

适用于桌宠、QQ 机器人、群聊守护、自动化触发等场景。

- 输入可能是连续事件流，而不是每条都要回复
- 决策模型先判断：
  - 是否回复
  - 是否调用工具
  - 是否执行动作（如发送消息、更新状态）
- 决策通过后才进入 `Reply` 生成

## 6. 发行形态：应用版 + 无头版

### 6.1 应用版（App）

- 基于 Tauri
- 当前以桌面形态运行，后续可扩展移动端形态
- 与无头版共享同一套核心逻辑和前端页面结构

### 6.2 无头版（Headless）

- 运行形态：常驻 Rust 服务进程，监听端口
- 提供 Web 配置页（页面与应用版高度一致）
- 适用于无桌面环境：服务器、NAS、容器、云主机
- 同时暴露管理 API 与运行时 API（可供外部系统集成）

建议最小接口：

- `GET /health`：健康检查
- `GET /api/bootstrap`：前端初始化数据
- `GET/POST /api/providers`：提供商配置
- `GET/POST /api/models`：模型配置
- `GET/POST /api/agents`：智能体配置

### 6.3 同构前端策略（重点）

- 前端仅维护一套 `apps/web/console` 代码
- 应用版通过 Tauri invoke 调用后端
- 无头版通过 HTTP API 调用后端
- 通过“运行时客户端适配器”切换调用方式，保证页面能力一致

### 6.4 无头版安全基线

- 默认只监听 `127.0.0.1`
- 提供 Token 鉴权或账号鉴权
- 支持反向代理后启用 HTTPS
- 敏感配置（Key）加密存储，不在前端明文回显

## 7. 后端建议职责划分（Rust）

- `provider` 模块：Provider 配置校验、Key 池与健康检查
- `model` 模块：模型配置、能力声明、参数模板
- `adapter` 模块：云端/本地协议适配器
- `agent` 模块：人格模板、提示词组装、槽位路由
- `decision` 模块：事件判断与动作触发策略
- `runtime` 模块：请求调度、失败重试、负载均衡
- `storage` 模块：本地配置持久化（建议 SQLite + 敏感字段加密）
- `host-app`：应用宿主绑定层（当前为 Tauri）
- `host-headless`：HTTP 宿主绑定层（建议 `axum`）

## 8. 前端建议页面结构

导航：

- 提供商设置
- 模型设置
- 智能体设置

智能体设置页建议分区：

- 基本人格
- 模型槽位（组件/工具/回复/决策）
- 行为规则
- 高级参数

交互原则：

- 优先“可用默认值”
- 高级参数折叠显示
- 关键字段实时校验（API URL、Key、模型能力匹配）
- 通过能力标签防止错误绑定（如把 TTS 绑定到视觉槽位）

## 9. V0 -> V3 演进路线

V0（当前）：

- 工程骨架
- 设置页 UI 骨架
- 基础后端命令接口

V1：

- 配置持久化
- Provider / Model / Agent 基本 CRUD
- 云端模型联通与连通性测试
- 多 Key 负载策略（轮询/失败切换）
- 无头版最小可用（监听端口 + Web 设置页 + 基础鉴权）

V2：

- 本地模型适配层（Ollama / OpenAI 兼容网关优先）
- 模型能力声明与自动路由
- 语音链路（ASR + TTS）最小闭环
- 工具系统（检索、函数调用、RAG）
- 同构前端完善（应用/无头页面一致性）

V3：

- 决策模型完整上线（桌宠/QQ 机器人场景）
- 常驻事件循环与动作执行器
- 多智能体协作与配置导入导出
